# Evidence Standards

Canonical product page: <https://justlay.me/grey-mirror/evidence-standards>

Grey Mirror uses an evidence-first framing: reports should explain why a signal was produced, not simply label a relationship from a few emotionally charged messages.

## What Counts As Evidence

Evidence can include:

- Message-level signals, such as explicit repair attempts or future planning language.
- Window-level signals, such as changes between early and recent periods.
- Participant-level signals, such as effort balance or turn-taking asymmetry.
- Timeline signals, such as long response gaps or repeated cycles.
- Metric-level signals, such as counts, rates, deltas, and confidence labels.

## Evidence Windows

An evidence window is a bounded portion of the timeline used to support a claim. The goal is not to expose unnecessary private text. The goal is to show enough context for the user to understand why a score, label, or change was produced.

## Confidence Labels

Confidence should be lower when:

- The upload is incomplete.
- Participant mapping is ambiguous.
- The relevant message count is low.
- Timestamps are missing or inconsistent.
- The pattern depends on tone, sarcasm, or off-platform context.
- Evidence exists but is too sparse for a strong claim.

## Insufficient Evidence

If a report does not have enough support for a metric, the honest output is an insufficient-evidence state. Grey Mirror should avoid filling gaps with invented certainty.

## Access Control

Public documentation can describe evidence concepts, but user-specific evidence should stay behind the product access model. This repository does not publish private reports, uploads, model artifacts, or message examples from real users.
